用 React Native 写了一个 RSS 阅读器

发表于 2019-04-12
最近用 React Native 写了一个 RSS 阅读器，兼容 Android 和 iOS，虽然界面比较粗糙，还是实现基本实现RSS订阅、阅读功能，因为使用雅虎的YQL间接向接口请求数据，还可以访问到一些被墙的RSS源。

update_2017_08_09

YARRD 是我刚接触 react native 时候的练手项目，如今看来，代码上有许多不规范的地方，暂时也没有精力重构。而写这个项目过程中最有意思发现，是利用雅虎的YQL服务，可以绕过GFW的封锁访问到一些网站。

最近用 React Native 写了一个 RSS 阅读器，兼容 Android 和 iOS，虽然界面比较粗糙，还是实现基本实现RSS订阅、阅读功能，因为使用雅虎的YQL间接向接口请求数据，还可以访问到一些被墙的RSS源。给它起了个名字 YARRD，取自 yet a rss reader 的缩写。

YARRD 介绍

YARRD 的 UI 基本参考 digg ，总共有四个界面：

  1. FeedBoxes，列表展示所有的 RSS 订阅源，也是应用的首页:

    订阅了的 RSS 源都会享受在这里，如果想删除某个RSS，只要按住它向左滑动，就会显示删除按钮。

  2. FeedSearch，搜索 RSS 的界面, 搜索结果只能显示 20 条，每条结果会显示RRS源的标题，最后更新时间，RSS源简介，有些源的“最后更新时间”是很久以前的，很可能就是停止更新了：

  3. FeedList，查看 RSS 源输出内容的界面,支持下来刷新、上拉加载，每次加载10条新内容:

  4. FeedDetails，查看 RSS 文章的界面，文章最后跟着一个按钮，点击可以跳转到浏览器访问文章的原始链接。

    每个RSS源输出的内容长短不同，有的是全文输出，有的只有寥寥几句摘要。如果RSS输出的只有摘要，那只能通过访问原始链接来阅读全文。

    对于一个阅读器来说，能在其中展示订阅源的全文当然是最好的，遗憾的是，大多数RSS源只输出文章的摘要（大概他们需要把读者引到自己的网站，增加流量的原因）。这种情况在遇到一些被屏蔽的网站时更是无奈。比如纽约时报中文网的RSS，在这个阅读器里只能读到简短的摘要(如上图)。而同样是被屏蔽的新闻网站端传媒，因为RSS是全文输出（如下图），所以可以在 FeedDetails 读到全文。

使用的API：

  1. Search 页面使用的搜索引擎是 feedly.com 的搜索API： https://feedly.com/v3/search/feeds
    还有一个digg的搜索API可以备选，不过没有在YARRD中使用。
    从我使用的体验看，feedly 的搜索结果还是很精准的，基本能找到你想要的 RSS。

  2. RSS 订阅源的的数据是通过digg的API来获取的，因为这样可以让不同 RSS 返回同样的格式，方便在客户端渲染。
    比如，纽约时报中文网的RSS地址是:
    http://cn.nytimes.com/rss.html
    把它当中其中一个参数传入digg的获取RSS的API:
    http://digg.com/api/reader/feed.json
    参数示例如下：
    param = { feed_url:'http://cn.nytimes.com/rss.html', //RSS url show: all, //不知道做什么的 count:20, //返回条目 _:1485757069742 //时间戳 }
    最后的 request URL 是这样的：
    http://digg.com/api/reader/feed.json?feed_url=http%3A%2F%2Fcn.nytimes.com%2Frss%2Fnews.xml&show=all&count=20&_=1485757069742
    经过了digg的处理，不同的RSS源都会返回相同的json格式。

  3. 因为 digg 在国内经常无法正常访问，可以用雅虎的YQL间接请求数据。
    YQL 的 API 如下:
    https://query.yahooapis.com/v1/public/yql
    以上面提到的digg API请求为例，可以构造参数：

    最后的 request URL:
    https://query.yahooapis.com/v1/public/yql?q=select%20*%20from%20json%20where%20url%3D%22http%3A%2F%2Fdigg.com%2Fapi%2Freader%2Ffeed.json%3Ffeed_url%3Dhttp%253A%252F%252Fcn.nytimes.com%252Frss%252Fnews.xml%26show%3Dall%26count%3D20%26_%3D1485757069742%22&format=json

存在问题

  1. (2017-02-07 修复) 经过一番摸索，已经可以恰当地渲染 HTML 内容了。使用的是 React Native 的 WebView，注入 javascript 脚本来控制图片的显示尺寸，以及 WebView 的高度。目前还有一点瑕疵，页面渲染的速度有一点点慢。

    许多RSS源输出的内容里含有HTML标签，在 React Native 似乎没有无法很好地渲染。
    下面是一段摘自纽约时报中文的RSS信息，里面就含有<img>等标签：
    世界卫生组织本周警告各国警惕禽流感疫情。此前中国刚报告了两个死亡病例；春节期间人口流动和活禽运输增多，可能传播流感。9月以来，中国报告了225个病例。<p><img alt="2014年，在从内地进口的家禽中发现了H7N9病毒后，香港的工作人员在捕杀家禽。 Isaac Lawrence/Agence France-Presse — Getty Images" src="https://static01.nyt.com/images/2017/01/26/science/26FLU/26FLU-articleLarge.jpg" /></p> <p>Isaac Lawrence/Agence France-Presse — Getty Images</p> <p>2014年，在从内地进口的家禽中发现了H7N9病毒后，香港的工作人员在捕杀家禽。</p>
    这种情况下，使用React Native 的 WebView 并不能控制图片的尺寸，图片会直接显示原始的宽高，有一大截超出手机屏幕（如下图）：

    我目前使用的是第三方库 react-native-htmlview，虽然它无法渲染格式、图片，不过勉强使内容适应了手机屏幕的宽度，且当权宜之计吧。

  2. YQL 返回的数据不能和被查询的url返回的数据保持一致。
    以上述的digg API http://digg.com/api/reader/feed.json 为例,它还可以传入第5个参数：next_positon，用来确定起始的RSS feed。
    没有传入 next_position 时，digg 默认返回该RSS 源最近的20条feed。每条feed都带有它的发表时间的时间戳。
    假如第20条feed的时间戳是”1485215019”，把这个时间戳传入next_position,如下:
    param={ feed_url:'http://cn.nytimes.com/rss.html', //RSS url show: all, //不知道做什么的 count:20, //返回条目数量 _:1485757069742 //时间戳 next_positon:"1485215019" }
    那么，digg 将返回由1485215019这条feed起算的后20条feed，即最近的第20～40条。这样就可以实现加载更多的需求。
    然而YQL返回的数据并不理会 next_positon，无论你传入什么，它返回的仍然是最近二十条feed。
    无奈之下，我只能通过修改count来实现加载更多的条目，比如第一次加载是20条，第二次加载是40条……
    另外，一些经过 feedburner.com RSS源也不能获取到，比如http://feeds2.feedburner.com/programthink ，在digg可以获取到数据，通过YQL则只能得到 null 。
    或许 YQL 本身出了问题。从 YQL 陈旧的页面来看，雅虎似乎没有对这款产品投入太多精力。网上也有传言说，雅虎已经不再维护 YQL。

Roadmap

  1. 这个RSS只是粗暴地实现了一些简单的功能，目前有一个最大缺陷就是无法实现聚合阅读，所有的RSS阅读器都应该有一个 ALL 的选项，把所有的RSS信息按时间排列到一起。而 YAARD 目前只能挨个阅读RSS源。如何把信息聚合到一起，暂时没有想到比较好的解决思路。
  2. 如果能加入用户注册功能，那么用户已经订阅的RSS就可以跨设备共享。Google 的 firebase 提供了一个很好的后端，以后试试能不能接入。
  3. 像纽约时报中文网这种不输出全文、网站又被屏蔽RSS，理论上可以利用YQL，获取全文数据。
